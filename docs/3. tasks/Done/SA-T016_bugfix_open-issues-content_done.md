# SA-T016 — Исправить автозаполнение Open Issues из произвольного текста

## Customer-facing инкремент

После архивирования сессии поле Open Issues будет содержать только реальные нерешённые проблемы, а не stdout Python-скриптов, reasoning-комментарии ассистента или meta-текст из verify-archive.

---

## Баги

### BUG-OPEN-ISSUES-CONTENT — Open Issues содержит stdout Python-скрипта или conversational текст

**Симптом:** `sessions.open_issues` и секция `## Open Issues` в MD-экспорте содержат вывод типа:
- `"Удалено пустых сессий: 19 (осталось: 55)"` — stdout Python-скрипта очистки BUGS.md
- `"Контекст уже есть — T288..."` — ответ ассистента на вопрос
- `"Противоречие было в оригинале..."` — conversational фрагмент

**Root cause:** `extract_open_issues()` (или аналогичная логика) сканирует весь assistant-текст сессии в поиске "проблем". Stdout от запуска внешних скриптов через Bash tool попадает в tool result → assistant обрабатывает его и упоминает в своём ответе → парсер ошибочно захватывает как open issue.

**Повторялось:** 7+ раз, в каждой сессии с /bugs-to-task и /verify-archive исправляется вручную.

**Fix:**
Open Issues парсить ТОЛЬКО из явно структурированных секций — например, markdown-блоков с меткой `ISSUE:`, `WARN:`, `BUG:` или из dedicated `## Open Issues` секции, которую ассистент заполняет явно. Не захватывать произвольный assistant-текст.

Вариант 1 (минимальный): Если `open_issues` — строка и содержит паттерн `Удалено .* сессий` или `осталось:` — очищать до `[]`.

Вариант 2 (правильный): `extract_open_issues()` должна работать только с явно размеченными блоками в тексте, а не с free-form prose. Если явной разметки нет — возвращать `[]`.

---

### BUG-OPEN-ISSUES-REASONING-TEXT — Open Issues содержит reasoning/meta-текст ассистента

**Симптом:** Open Issues содержит внутренние reasoning-комментарии verify-archive:
- `"3. Events: tests_run — ложный"`
- `"Теперь фикшу export MD..."`
- `"archive-session не запускался, но в IDE открыт файл..."`
- `"3. ## Open Issues — содержит внутренний reasoning-комментарий"`

**Root cause:** verify-archive генерирует текст вида "Нашёл баг X, фикшу..." прямо в ответе ассистента. Парсер open_issues захватывает весь assistant-текст сессии, включая этот reasoning. Вторичный эффект: meta-описание самого бага попадает в open_issues следующей архивации.

**Повторялось:** 5+ раз.

**Fix:**
Тот же, что BUG-OPEN-ISSUES-CONTENT: парсить open_issues только из явно маркированных блоков. Reasoning-текст verify-archive не должен проходить фильтр.

Дополнительно: verify-archive может явно закрывать open_issues в конце работы командой типа `UPDATE sessions SET open_issues = '[]' WHERE session_id = ?` если не нашёл реальных проблем.

---

## Scope

- Изменить логику `extract_open_issues()` в `session_archive.py`: парсить только явно размеченный контент
- Добавить фильтр-эвристику: если open_issues содержит stdout-паттерны (`Удалено`, `осталось:`) — обнулять
- Тесты: убедиться что reasoning-текст не попадает в open_issues

## Out of scope

- Изменение формата экспорта MD
- Добавление явной разметки в промпты verify-archive и bugs-to-task

---

## Как протестировать

1. Запустить /bugs-to-task в тестовой сессии → архивировать → проверить `open_issues` в SQLite
2. Запустить /verify-archive → архивировать → проверить что reasoning-текст не попал в open_issues
3. Запустить обычную сессию без явных проблем → `open_issues` должен быть `[]`

## Критерии приёмки

1. После /bugs-to-task сессии `open_issues = []` (нет stdout скрипта)
2. После /verify-archive сессии `open_issues = []` (нет reasoning-текста)
3. Если в сессии есть реальная проблема явно описанная ассистентом в структурированном виде — она попадает в open_issues
