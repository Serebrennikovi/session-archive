# SA-T055 — Откаченные пользователем правки попадают в Diffs

## Проблема

Дифы записывают ВСЕ Edit/Write вызовы из сессии, включая промежуточные версии, которые пользователь откатил. В результате в архиве содержатся дифы, которые не соответствуют финальному состоянию файлов.

### Воспроизведение (2026-04-08, session 23eb9256)

1. В ходе сессии были созданы CSS-стили `.cmp-duo`, `.cmp-card`, `.cmp-card--bad`, `.cmp-card--good` и соответствующий HTML (duo-карточки сравнения)
2. Пользователь откатил эти изменения ("дуо карточки я откатил")
3. Позже были применены другие правки к таблице (2-column cmp-table)
4. В Diffs присутствуют ОБА варианта: и cmp-duo (строки 369-380), и cmp-table 2-column (строки 420-436)
5. Финальное состояние файла содержит только cmp-table вариант

### Влияние

Архив содержит "мусорные" дифы, которые вводят в заблуждение при ретроспективном анализе. Читатель не может отличить финальные правки от откаченных.

### Вероятная причина

Парсер записывает каждый tool_use Edit/Write как отдельный diff-блок без отслеживания, были ли эти изменения впоследствии отменены.

### Воспроизведение 2 (2026-04-13, session 946bb737, sales)

Также проблема с **пропущенными** edit-дифами (обратная сторона):
1. Write создал index_v3.html (896 строк) — записан как write_snapshot
2. 4 последующих Edit вызова (hero padding, pains padding, nav restructure, JS cleanup) — **не записаны** в diffs
3. Diff в архиве показывает начальное состояние, НЕ финальное
4. Один из edit-ов (`--sp-xl` → `48px`) был впоследствии заменён на `--sp-lg` — промежуточный edit не должен быть в архиве
5. Ручной фикс: добавлены 4 edit-дифа в export .md

**Связанная проблема**: парсер не записывает Edit tool_use как отдельные diff-блоки после Write (только write_snapshot).

## Рекомендация по фиксу

1. **Минимальный**: пометить в diff-блоке если один и тот же файл правился многократно: `[edit 3/7 — file was edited 7 times total]`
2. **Средний**: для файлов с ≥3 Edit-вызовами — сворачивать в один aggregate diff (first old_string → last new_string)
3. **Идеальный**: сравнивать финальное состояние файла (последний Write/Edit) с начальным (первый Read или git state) и генерировать один consolidated diff
