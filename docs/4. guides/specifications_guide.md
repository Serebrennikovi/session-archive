## 1. Назначение спецификации

Спецификация — это **разработческий документ**, который отвечает на вопросы:

* **Что именно нужно построить** — какой функционал, интерфейс, поведение
* **Зачем это важно сейчас** — какой инкремент получает пользователь или система
* **Какие границы реализации** — что входит и что явно не входит в scope
* **Как это устроено** — архитектура, структура, интеграции
* **Как проверяем результат** — критерии приёмки (DoD)

---

## 2. Типы спецификаций

### 2.1. Функциональные спецификации

**Назначение:** описать новую функциональность, API, CLI команды, изменения в БД.

**Содержит:**
* Архитектуру изменений
* Изменения в DB схеме (новые таблицы, колонки, индексы)
* Новые CLI команды или изменения существующих
* Модели данных
* Acceptance Criteria

**НЕ содержит:**
* Детальный код реализации (это в задачах)
* Подробные бизнес-обоснования (это в BR)

### 2.2. Технические спецификации (рефакторинг, оптимизация)

**Назначение:** описать техническое улучшение без новой функциональности.

**Содержит:**
* Текущее состояние и проблему
* Предлагаемые изменения
* Ожидаемые метрики улучшения
* Риски

## 3. Структура спецификации

### Шаблон имени файла

> См. [doc_conventions.md](doc_conventions.md) — единый формат для всех спецификаций

```
SA-SNN_<snake_case_название>.md
```

Примеры: `SA-S01_fts_search.md`, `SA-S02_web_dashboard.md`

---

### 3.1. Заголовок

```markdown
# Спецификация: <Название фичи>

**ID:** SA-SNN
**Статус:** draft / active / done / deprecated
**Версия:** X.X
**Дата:** YYYY-MM-DD
```

---

### 3.2. Цель

1–2 абзаца:
* Что добавляем/меняем
* Какую задачу решаем
* Почему это важно сейчас

**Пример:**
```
FTS (full-text search) позволит искать по тексту всех сессий напрямую из CLI.
Это избавит от необходимости выгружать данные и grep-ать по markdown экспортам.
```

---

### 3.3. Scope

#### Входит:
* Конкретный перечень функций, команд, изменений

#### Не входит / Out of scope:
* Всё, что точно не делаем на этом этапе

**Пример:**
```markdown
### Входит:
- Включение FTS5 в SQLite для таблицы session_messages
- Команда `search <query>` в session_archive.py
- Базовое ранжирование по релевантности

### Не входит:
- Веб-интерфейс поиска (будет в SA-S02)
- Поиск по артефактам (следующая спека)
- Fuzzy matching
```

---

### 3.4. Архитектура и структура

Описать:
* Какие компоненты создаются или изменяются
* Как это встраивается в существующую систему
* Диаграммы взаимодействия (если нужно, текстом)

---

### 3.5. Изменения в БД

Описать:
* Какие таблицы/колонки создаются/изменяются
* Миграция через `_ensure_column()` (если добавляем колонку) или `CREATE TABLE IF NOT EXISTS`

**Пример:**
```sql
-- FTS виртуальная таблица
CREATE VIRTUAL TABLE IF NOT EXISTS sessions_fts
USING fts5(id, summary, content=sessions);

-- Триггеры для поддержания актуальности FTS индекса
CREATE TRIGGER sessions_ai AFTER INSERT ON sessions BEGIN
  INSERT INTO sessions_fts(rowid, id, summary) VALUES (new.rowid, new.id, new.summary);
END;
```

---

### 3.6. CLI интерфейс (если применимо)

Описать новые команды или изменения существующих.

**Пример:**
```
python3 session_archive.py search <query> [--limit N] [--project <name>]
```

**Аргументы:**
- `query` — поисковый запрос (поддерживает FTS5 синтаксис)
- `--limit N` — максимум результатов (дефолт: 20)
- `--project <name>` — фильтр по проекту

**Выходной формат:**
```json
[{"session_id": "...", "repo_name": "...", "snippet": "...", "rank": 1.23}]
```

---

### 3.7. Acceptance Criteria / DoD

Список критериев, по которым можно считать фичу готовой.

**Пример:**
```
- [ ] `python3 session_archive.py search "deploy"` возвращает результаты
- [ ] Результаты ранжированы по релевантности
- [ ] FTS индекс обновляется при добавлении новых сессий
- [ ] Поиск работает по summary и тексту сообщений
- [ ] Команда `stats` показывает размер FTS индекса
```

---

### 3.8. Тест-план

Сценарии тестирования вручную.

**Пример:**
```
1. Заархивировать тестовую сессию с известным текстом
2. Запустить `search <слово из сессии>` — ожидается: сессия в результатах
3. Запустить `search <несуществующее_слово>` — ожидается: пустой список
4. Проверить `stats` — FTS индекс показывается
```

---

### 3.9. Зависимости и интеграции

Указать:
* Что должно быть готово до начала
* Какие другие спецификации зависят от этой

---

### 3.10. Риски и ограничения

Зафиксировать потенциальные проблемы.

**Пример:**
```
- FTS5 не поддерживается в SQLite < 3.20. Проверить версию Python.
- Размер индекса может значительно вырасти на больших БД.
```

---

### 3.11. Связанные документы

```markdown
- Архитектура: [SA-architecture.md](../SA-architecture.md)
- Зависит от: (нет)
- Блокирует: SA-S02 (веб-интерфейс)
```

## 4. Принципы хорошей спецификации

### 4.1. Строгое разделение "что строим" и "как проверяем"
Scope ≠ Acceptance Criteria. Scope описывает функционал, DoD — критерии готовности.

### 4.2. Reuse-first подход
Всегда писать:
* Что переиспользуем из `_ensure_column()` / `_init_schema()`
* Что рефакторим
* Что удаляем/заменяем

### 4.3. Без зависимостей (stdlib only)
Любые новые зависимости в `session_archive.py` — отдельное обоснование в спеке.
`analyze.py` может добавлять опциональные зависимости с try/except.

### 4.4. Одна спека = один инкремент
После реализации должно быть что-то новое и работающее.

### 4.5. Стиль: точный, сухой, инженерный
Без лирики. Конкретные решения и чёткие критерии.
