# Гайд: декомпозиция спецификаций на задачи

**Версия:** 1.2
**Дата создания:** 26 декабря 2025
**Дата обновления:** 11 января 2026
**Статус:** Действует

---

## Цель

Установить единый подход к декомпозиции спецификаций на задачи, обеспечивающий:
- Проверяемые инкременты на каждом шаге
- Управляемый размер задач (до 50% контекста AI)
- Чёткую связь между спекой, задачами и customer-facing результатами
- Прозрачность прогресса и зависимостей

---

## 1. Принципы декомпозиции

### 1.1. Декомпозиция по инкрементам, а не по слоям

❌ **Плохо** (по слоям):
```
T01 — Модели данных
T02 — API endpoints
T03 — UI компоненты
T04 — Стили
```

✅ **Хорошо** (по инкрементам):
```
T01 — Пользователь может видеть список постов
T02 — Пользователь может создать новый пост
T03 — Пользователь может лайкнуть пост
```

**Почему:**
- Каждая задача даёт проверяемый результат
- Можно показать прогресс заказчику после каждой задачи
- Проще тестировать и принимать работу

### 1.2. Размер задачи: до 50% контекста AI

**Практические критерии:**
- 1-3 компонента/файла
- 1-2 модели данных или API endpoints
- Реализуемо за 1-2 сессии работы

**Если задача больше:**
- Разбить на подзадачи по инкрементам
- Выделить техническую подготовку в отдельную задачу

### 1.3. Каждая задача = проверяемый инкремент

**Обязательно:**
- Задача должна давать результат, который можно **показать** и **проверить**
- Результат должен быть **ценным**

**Допустимые исключения:**
- Технический рефакторинг (но с явным обоснованием)
- Миграции данных (с проверяемым результатом)

### 1.4. Интеграции с внешними API: верификация по документации

**При написании задач, связанных с интеграцией внешнего API:**

1. **Перед написанием кода** — найти и прочитать актуальную документацию API (не полагаться на примеры из архитектурных документов или старых задач)
2. **Зафиксировать в задаче** ссылки на использованные страницы документации
3. **Сохранить справочник API** в `docs/5. unsorted/` для быстрого доступа
4. **Верифицировать формат запросов/ответов** — эндпоинты, поля, коды ошибок, rate limits

## 2. Подходы к декомпозиции

### 2.1. По user journey (для UX-тяжёлых фич)

**Пример:** Онбординг пользователя

```
T01 — Экран приветствия и выбор интересов
T02 — Заполнение профиля (имя, аватар, должность)
T03 — Подписка на рекомендованных пользователей
T04 — Первый пост в ленте
```

### 2.2. По техническим слоям (для API-тяжёлых фич)

**Пример:** Система сообщений

```
T01 — API для получения списка чатов
     Инкремент: Пользователь видит список своих чатов

T02 — API для отправки сообщений
     Инкремент: Пользователь может отправить сообщение

T03 — Real-time обновления
     Инкремент: Сообщения появляются мгновенно
```

### 2.3. По независимым фичам (для параллельной работы)

**Пример:** Улучшения аналитики

```
T01 — Отчёт по времени работы над проектами
T02 — Граф зависимостей задач
T03 — Экспорт в CSV
```

Эти задачи независимы и могут выполняться параллельно.

### 2.4. Выделение технической подготовки

**Когда нужно:**
- Рефакторинг перед новой фичей
- Миграция данных
- Обновление зависимостей

**Пример:**
```
T01 — Рефакторинг: вынести парсинг событий в отдельный модуль
     Обоснование: Упрощает добавление новых event_type в T02-T04

T02 — Поддержка event_type: pr_reviewed
     Инкремент: PR ревью фиксируется в БД
```

## 3. Нумерация задач (TNN)

### 3.1. Принципы

- Нумерация **сквозная** по всему проекту (SA-T001, SA-T002, SA-T003...)
- Номера выдаются в порядке создания
- **ЗАПРЕЩЕНО** добавлять буквы к номерам (SA-T001a — запрещено)

### 3.2. Как определить следующий номер

```bash
find "docs/3. tasks" -name "SA-T*.md" | grep -oE 'T[0-9]+' | grep -oE '[0-9]+' | sort -n | tail -1
```

### 3.3. Параллельные задачи

Если задачи можно делать параллельно:
- Нумеровать последовательно (SA-T001, SA-T002, SA-T003)
- В SA-HANDOFF.md указать, что можно делать параллельно

## 4. Структура задачи

### 4.1. Шапка (обязательно)

```markdown
**Дата:** YYYY-MM-DD
**Статус:** draft | in progress | done
**Спецификация:** docs/2. specifications/SA-SNN_<имя>.md

# SA-TNN_sNN_<название> — <Краткое описание>
```

### 4.2. Customer-facing инкремент (обязательно)

```markdown
## Customer-facing инкремент

Пользователь может <действие> и получает <результат>.
```

### 4.3. Scope / Out of scope (обязательно)

```markdown
## Scope
- Что делаем

## Out of scope
- Что НЕ делаем в этой задаче
```

### 4.4. Как протестировать (обязательно)

```markdown
## Как протестировать

1. Запустить python3 analyze.py X
2. Проверить, что Z отображается
```

### 4.5. Критерии приёмки (обязательно)

```markdown
## Критерии приёмки

1. Команда X выводит Y
2. Запись в БД содержит поле Z
3. Markdown экспорт содержит секцию W
```

## 5. Жизненный цикл задачи

### 5.1. Создание (draft)

**Структура папок:**
```
docs/3. tasks/
├── Done/                         # Завершённые задачи
└── SA-TNN_sNN_<название>.md      # Активные задачи в корне
```

**Правила:**
1. Создать файл `docs/3. tasks/SA-TNN_sNN_<название>.md`
2. **Добавить задачу в SA-HANDOFF.md** в соответствующую секцию
3. **Добавить ссылку в спеку** (SA-SNN_*.md) в раздел «Задачи»

### 5.2. В работе (in progress)

Обновить статус в файле и SA-HANDOFF.md

### 5.3. Завершение (done)

**После акцепта:**
1. Статус → `done`
2. Переместить в `docs/3. tasks/Done/SA-SNN_*_done/SA-TNN_*_done.md`
3. Обновить SA-CHANGELOG.md и SA-HANDOFF.md

### 5.4. Отмена (cancelled)

1. Суффикс → `_cancelled.md`
2. Указать причину отмены
3. Переместить в Done

## 6. Управление зависимостями

### Формат в SA-HANDOFF.md

```markdown
## Следующие шаги

### Приоритет 1: FTS поиск (SA-S01)

1. SA-T001 — Индексирование текста
   - **Блокирует:** SA-T002

2. SA-T002 — Search API
   - **Требует:** SA-T001

3. SA-T003 — Web UI для поиска
   - **Требует:** SA-T002
   - **Можно параллельно с:** SA-T004
```

## 7. Чек-лист: готова ли задача?

### Для draft-задачи:

- [ ] Шапка (дата, статус, ссылка на спеку)
- [ ] Customer-facing инкремент
- [ ] Scope / Out of scope
- [ ] Как протестировать
- [ ] Критерии приёмки
- [ ] Размер до 50% контекста AI
- [ ] **Добавлена в SA-HANDOFF.md**
- [ ] **Добавлена ссылка в спеку**

### Для done-задачи:

- [ ] Статус `done`, дата завершения
- [ ] Критерии приёмки выполнены
- [ ] Файл перемещён в Done
- [ ] SA-HANDOFF.md и SA-CHANGELOG.md обновлены

---

## 8. Частые ошибки

| Ошибка | Решение |
|--------|---------|
| Задача слишком большая | Разбить по инкрементам |
| Нет проверяемого результата | Переформулировать инкремент |
| Дублирует спеку | Добавить конкретику: файлы, шаги |
| Неясные зависимости | Указать в SA-HANDOFF.md |
| SA-T001a вместо SA-T002 | Перенумеровать со следующего свободного номера |
